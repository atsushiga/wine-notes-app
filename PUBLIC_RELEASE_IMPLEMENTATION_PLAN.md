# Public Release Implementation Plan

この文書は、Wine Notes を一般公開または招待制ベータとして提供するために必要な実装内容をまとめたものです。

公開方針の前提:

- まず招待制ログイン必須のクローズドベータとして公開する。
- 個人のテイスティングノート、画像、AI生成履歴は本人のみアクセス可能にする。
- AI/STT/画像生成/アップロードはコストが発生するため、認証、容量制限、利用量上限、監視を必須にする。

## 1. 認証必須化

目的: 個人データ画面、API、AI処理を未ログインユーザーから使えない状態にする。

実装内容:

- `src/middleware.ts` を Next.js 16 の `src/proxy.ts` に移行する。
- 保護対象を `/`, `/tasting-notes`, `/statistics`, `/settings`, `/wines/*`, `/ai-explainer/*` に拡張する。
- `/login`, `/signup`, `/set-password`, `/auth/callback`, `/offline`, 静的アセット、PWA関連ファイルは公開のままにする。
- API Route と Server Action の共通認証ヘルパーを追加する。
- `/api/submit`, `/api/stt`, `/api/upload`, `/api/upload-url`, AI系Server Actionの冒頭でログインユーザーを必須にする。
- `/api/submit` は未ログインなら `401` を返し、保存時は必ず `user_id` を付与する。

完了条件:

- 未ログインで主要画面にアクセスすると `/login` にリダイレクトされる。
- 未ログインで保護APIを呼ぶと `401` が返る。
- 保存、更新、削除、AI、STT、アップロードはログインユーザーのみ実行できる。

## 2. 画像とRLSの本人限定化

目的: ワイン画像と画像メタデータを本人以外が読めないようにする。

実装内容:

- `wine_images` の公開読み取りpolicyを削除するmigrationを追加する。
- `wine_images` のselect/insert/update/delete policyを `tasting_notes.user_id = auth.uid()` ベースに統一する。
- `/api/images/[...path]` で認証を必須にし、対象GCS keyがログインユーザーのノート画像またはAI解説画像に紐づく場合だけ返す。
- 新規アップロードのGCS pathを `uploads/{userId}/YYYY/MM/...` に変更する。
- 既存画像の移行方針を決める。ベータ前であれば既存画像の削除または再アップロードでもよい。
- サービスロールを使う処理では、RLSを迂回する前にアプリ側で所有者チェックを行う。

完了条件:

- 他ユーザーの画像URLを直接叩いても `403` または `404` になる。
- Supabase上でもRLSで他ユーザーの `wine_images` が読めない。
- 新規アップロード画像はユーザー単位のpathに保存される。

## 3. レート制限、容量制限、AI利用上限

目的: 公開後のコスト暴走と乱用を防ぐ。

実装内容:

- Supabaseに `usage_events` テーブルを追加する。
- ユーザー単位、アクション単位、日次単位で利用回数と利用量を記録する。
- サーバー共通の利用量チェックヘルパーを追加する。
- AI解説、AI画像生成、ワインラベル解析、AI深掘り検索、STT、画像アップロード、サインアップメール送信に上限を設定する。
- `/api/stt` で音声ファイルのMIME type、サイズを検証する。
- `/api/upload` と `/api/upload-url` で画像MIME type、サイズ、拡張子を検証する。
- 上限超過時は `429` を返し、UI側で再試行可能なエラーメッセージを表示する。

初期上限案:

- AI解説生成: 10回/日
- AI画像生成: 3回/日
- AI深掘り検索: 20回/日
- ワインラベル解析/補正: 20回/日
- STT: 30回/日、1ファイル25MBまで
- 画像アップロード: 30回/日、1ファイル10MBまで
- サインアップメール: IPまたはメール単位で5回/日

完了条件:

- 高コスト処理はすべて上限チェックを通る。
- 上限超過時に処理が実行されない。
- 利用状況をSQLで確認できる。

## 4. 依存関係、公開設定、セキュリティ更新

目的: 既知脆弱性と公開設定の残骸を減らす。

実装内容:

- `next`、`@google-cloud/storage`、`postcss`、地図関連依存を更新する。
- `next.config.ts` から一時ngrokドメインを削除する。
- GCS CORS設定から一時ngrokドメインを削除する。
- `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy` などのセキュリティヘッダーを追加する。
- 必要に応じてCSPを導入できる下地を作る。
- `npm audit --omit=dev` のcritical/highを解消するか、残リスクを記録する。
- `npm run lint` の警告を可能な範囲で削減する。

完了条件:

- `npm run build` が成功する。
- `npm run lint` が成功する。
- `npm audit --omit=dev` でcritical/highが解消されるか、公開判断可能な残リスクが文書化される。
- 本番設定に開発用ドメインが残っていない。

## 5. 法務、信頼性、アカウント運用導線

目的: 一般ユーザーが安心して使える最低限の公開情報とデータ削除導線を揃える。

実装内容:

- `/terms` に利用規約を追加する。
- `/privacy` にプライバシーポリシーを追加する。
- `/contact` に問い合わせ導線を追加する。
- 設定画面に退会/データ削除導線を追加する。
- 退会時に `profiles`, `tasting_notes`, `wine_images`, `ai_explanations`, GCS画像を削除または削除予約できるようにする。
- AI生成情報の免責を追加する。
- 画像、音声、ラベル情報がAI処理されることを明記する。
- 飲酒に関する注意文を追加する。
- PWA名、メタデータ、説明文を正式名称に更新する。

完了条件:

- フッターまたは設定から規約、プライバシー、問い合わせ、退会に到達できる。
- 退会処理が本人確認後に実行され、個人データが削除される。
- AI生成内容が参考情報であることがUI上で明示される。

## 6. テスト、監視、ベータ運用

目的: 公開後に壊れたことを検知し、回復できる状態にする。

実装内容:

- Playwrightで主要フローのE2Eテストを追加する。
- 認可テストとして、他ユーザーの詳細ページ、画像、APIにアクセスできないことを確認する。
- GitHub Actionsで `npm run lint`, `npm run build`, E2Eを実行する。
- Sentryなどのエラー監視を追加する。
- AI/API利用量を確認できるSQL viewまたは簡易管理用ドキュメントを追加する。
- Supabaseバックアップ、GCS lifecycle、ログ保持方針を文書化する。
- クローズドベータのチェックリストを追加する。

最低限のE2E対象:

- ログイン
- ワイン登録
- 画像アップロード
- 一覧表示
- 詳細更新
- 削除
- AI解説生成の上限チェック
- 他ユーザーのデータアクセス拒否

完了条件:

- CIで主要チェックが自動実行される。
- 本番エラー、API失敗、AIコスト、アップロード量を追える。
- ベータ開始前チェックリストが整備されている。

## 次点で必要な改善

公開前後で追加検討すべき項目:

- パスワードリセット導線のUI整備。
- メールアドレス変更時の確認フローの明確化。
- エラー画面、空状態、ローディング状態の品質向上。
- READMEを現行構成に合わせて更新し、未使用のGoogle Sheets/Notion説明を削除する。
- `robots.txt`, `sitemap.xml` の追加。
- サポート用の運用手順書を追加する。
- AI生成画像と出典付き画像の表示ルールを明確化する。
- データエクスポート機能を追加する。
- 管理者用の利用量確認、ユーザー停止、問い合わせ確認の運用を検討する。
- アクセシビリティ確認を行う。
- モバイルPWAとしてのインストール、オフライン時の挙動を検証する。
